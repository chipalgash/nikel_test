# RAG-система для анализа технических документов Норникель

## 🎯 Описание задачи

Разработка RAG (Retrieval-Augmented Generation) системы для ответов на вопросы по техническим документам с требованиями:
- **Короткие ответы** с ссылками на источники
- **Open-source модели** (без коммерческих лицензий)
- **Эффективная предобработка** данных и чанкинг
- **Качественный RAG пайплайн**

## 🔬 Экспериментальный процесс и решения

### 1. Предобработка данных

#### Попробованные подходы:
- ❌ **Извлечение изображений** - изначально планировалось обрабатывать таблицы и схемы из изображений
- ✅ **Фокус на тексте** - решили сосредоточиться на текстовом контенте для надежности
- ✅ **Два режима обработки таблиц**:
  - Детальный: построчная обработка с названиями столбцов
  - Простой: объединение через разделитель `|`

#### Финальное решение - функция `read_docx_text()`:
```python
def read_docx_text(docx_path: str) -> str:
    """Извлекает текст из DOCX включая таблицы"""
    document = Document(docx_path)
    parts = []
    
    # Параграфы
    for paragraph in document.paragraphs:
        if paragraph.text.strip():
            parts.append(paragraph.text.strip())
    
    # Таблицы (простой режим)
    for table in document.tables:
        for row in table.rows:
            row_text = [cell.text.strip() for cell in row.cells if cell.text.strip()]
            if row_text:
                parts.append(" | ".join(row_text))
    
    return "\n".join(parts)
```

### 2. Чанкинг

#### Экспериментированные параметры:
- **Размер чанка**: пробовали 800, 1000, **1200** (финальный), 1500 символов
- **Перекрытие**: тестировали 100, 150, **200** (финальный), 300 символов
- **Стратегия**: RecursiveCharacterTextSplitter для сохранения семантики

#### Финальные настройки:
```python
def split_text_into_chunks(text: str, max_chars: int = 1200, overlap: int = 200):
    splitter = RecursiveCharacterTextSplitter(
        chunk_size=max_chars,
        chunk_overlap=overlap,
    )
    return splitter.split_text(text)
```

**Обоснование выбора:**
- 1200 символов - оптимальный баланс между детализацией и контекстом
- 200 символов перекрытия - сохраняет связность между чанками

### 3. Выбор индекса и поиска

#### Попробованные подходы:
- ❌ **Только FAISS** - хорошо для семантического поиска, слаб на точных терминах
- ❌ **Только BM25** - отлично для ключевых слов, слаб на семантике
- ✅ **Гибридный подход** - FAISS + BM25 с ансамблевым ретривером

#### Финальная архитектура поиска:
```python
ensemble_retriever = EnsembleRetriever(
    retrievers=[
        self.bm25_retriever,  # Лексический поиск
        self.faiss_store.as_retriever()  # Семантический поиск
    ],
    weights=[0.5, 0.5]  # Равные веса после экспериментов
)
```

### 4. Модели и фреймворки

#### Выбранный стек:
- **Эмбеддинги**: `bge-m3` через Ollama (open-source)
- **Reranker**: `BAAI/bge-reranker-v2-m3` (open-source)
- **LLM**: `qwen3:14b` через Ollama (open-source)
- **Фреймворк**: LangChain + FAISS + Custom Pipeline

#### Обоснование выбора моделей:
- **bge-m3**: лучшее качество эмбеддингов для русского языка
- **qwen3:14b**: отличное понимание русского языка, структурированный вывод
- **bge-reranker-v2-m3**: точное переранжирование результатов

### 5. RAG пайплайн

#### Итоговая архитектура:
```
Запрос → [Гибридный поиск] → [Reranker] → [LLM генерация] → [Валидация]
   ↓           ↓                 ↓            ↓              ↓
 Embedding  FAISS+BM25     Переранжирование  Структурированный  Автоматическая
           (20 чанков)     (топ-10 чанков)   JSON ответ        оценка качества
```

#### Детальный pipeline в коде:
```python
def search(self, query: str) -> List[Chunk]:
    # 1. Гибридный поиск
    ensemble_retriever = EnsembleRetriever([bm25, faiss], weights=[0.5, 0.5])
    docs = ensemble_retriever.invoke(query)
    
    # 2. Преобразование в чанки
    chunks = self.cast_documents_to_chunks(docs)
    
    # 3. Переранжирование
    ranked_chunks = self.reranker.rerank_documents(query, chunks, numDocs=self.top_k)
    
    return ranked_chunks

def generate_answer(self, query: str, chunks: List[Chunk]):
    # 4. Формирование контекста
    context = '\n\n'.join([f'ID чанка: {chunk.chunk_id}\n{chunk.text}' for chunk in chunks])
    
    # 5. Генерация структурированного ответа
    response = self.llm.invoke(messages, format=LLMResponse.model_json_schema())
    
    # 6. Извлечение источников
    sources = self.get_answer_sources(chunks, response.chunks_id)
    
    return response.answer + '\n' + sources
```

## 🎯 Финальное решение

### Архитектура системы:
1. **Предобработка**: Извлечение текста из DOCX, включая таблицы
2. **Чанкинг**: 1200 символов с перекрытием 200
3. **Индексирование**: FAISS для семантики + BM25 для лексики
4. **Поиск**: Гибридный ансамбль с равными весами
5. **Переранжирование**: BGE reranker для точности
6. **Генерация**: Qwen3:14b с структурированным выводом
7. **Валидация**: Автоматическая оценка качества ответов

### Ключевые особенности:
- **Кэширование**: FAISS индексы сохраняются для быстрого перезапуска
- **Batch обработка**: Эффективная работа с reranker
- **Структурированный вывод**: JSON формат с ID использованных чанков
- **Автоматические источники**: Извлечение путей к документам
- **Детальное логирование**: Полная трассировка процесса

### Файловая структура:
```

```
